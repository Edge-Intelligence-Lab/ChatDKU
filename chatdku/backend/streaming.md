# Agent Streaming

This document explains the agent streaming workflow used in the separate agent pipeline architecture. The agent pipeline is decoupled from the Django backend to avoid unnecessary backend restarts and to improve modularity, scalability, and deployment flexibility.

---

# Working Mechanism

## Step 1: User Query

The workflow begins when a user submits a query $Q$ through the client application.

---

## Step 2: Django Backend

The `Django Backend` receives the query $Q$ when the endpoint:

```bash
POST /api/chat
```

is called.

The backend is responsible for:

- creating a chat response identifier
- managing session state
- forwarding the request to the agent pipeline
- exposing a streaming interface for retrieving responses

More information about the backend can be found here:

- [ChatDKU Backend Repository](https://github.com/Edge-Intelligence-Lab/ChatDKU-backend)

After receiving the request, the backend returns:

```json
{
    "chatId": "d20b0ebf-3c5e-403c-9d85-ae1ae8873as1",
    "sessionId": "b30581b0-9e44-4597-aec6-362b7ad223d7"
}
```

Where:

$$
\begin{aligned}
\textbf{chatId} &\rightarrow \text{Unique identifier for the generated response stream} \\
\\
\textbf{sessionId} &\rightarrow \text{Identifier representing the current conversation session}
\end{aligned}
$$

The client later uses $\textbf{chatId}$ to retrieve streamed responses from the agent pipeline.

---

### Request Flow

The high-level request flow is shown below:

```bash
POST /api/chat
        |
        v
{ chatId, sessionId }
```

## Step 3: FastAPI Backend and Django Expose
On sending `{chatId, sessionId}`, the backend initiates FastAPI based agent response. It also exposes an endpoint:

`/api/chat/{chatId}?sessionId={sessionId}`

The frontend can access the data via a `GET` request. Using the $\textbf{chatId}$ and $\textbf{sessionId}$ from above, we get

```bash
GET /api/chat/d20b0ebf-3c5e-403c-9d85-ae1ae8873as1?sessionId=b30581b0-9e44-4597-aec6-362b7ad223d7
```
The result are json streaming that looks like this:

```json
{
    "id": 1778491510817-0,
    "data": {"type": "reasoning", "stage": "start","content": "Agent started"}
}

{
    "id": 1778491528244-0
    "data": {"type": "reasoning", "stage": "Planner", "content": "1. Search for Duke Kunshan University's official policy on changing declared majors (procedure, deadlines, eligibility). 2. Look up any specific restrictions or requirements (e.g., GPA, course completion status) for switching majors. 3. Summarize the steps, including necessary forms or advisor approvals, for the student."}
}

...

{
    "id": 1778491536908-0
    "data": {"type": "reasoning", "stage": "Executor", "content": "VectorQuery: DKU procedure to change major deadline requirements advisor approval"}
}
...

{
    "id": 1778491550524-1
    "data": {"type": "end", "stage": "end", "content": ""}
}

```
Here, 

`id`: Represents a unique id fo rthe chunk streamed. It can be used to generate chunks from that particular point (useful in network reconnection)

`data`: Represents the payload generated by the agent. The payload has a fixed format i.e
```
{
    "type": The chunk segments
    "stage": Stage of generation
    "content": Chunk generation
}
```
Here,
$$
\text{type}  \in [\text{reasoning, chunk, end}] \\

\text{stage}  \in [\text{start, Planner, Executor, generation, end}] \\

$$
 
 Every response starts with

 ```json
{"type": "reasoning", "stage": "start", "content": "Agent started"}
 ```

 and ends with

 ```json
{"type": "end", "stage": "end", "content": ""}
 ```

 The `reasoning` types represent the intermediates in generation wheread `chunk` represents the final `output` generation. Each generation closes with `end`.
